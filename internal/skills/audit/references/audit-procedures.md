# 审计流程详情

> 本文件是 audit 的 Phase 2/3 详细流程和误报防御策略。

---

## Phase 2：验证方法详解

### 路径追踪

从报告的"问题行"出发，沿控制流追踪所有可达路径，确认是否真的会触发问题。

**检查清单**：
- 入口函数是否被调用？
- 条件分支是否可达？
- 变量值在执行路径上是否符合前置条件？

### 实际 diff

声称"不一致"的发现，逐行对比两个文件的实际内容。

**检查清单**：
- 用 Read 工具分别读取两个文件
- 逐行对比关键段落
- 确认差异是真实的还是审计 agent 的记忆偏差

### 设计意图查证

**检查清单**：
- 读取问题附近代码注释、测试名、fixture 和 hook 日志行为
- 必要时读取 CLAUDE.md / README / REFERENCE / docs 中的相关段落
- 文档只用于理解意图或发现候选矛盾；不能单独证明 C/H 级 bug
- 如果代码/测试明确说明"这是有意设计"，降级为 ⚠️ 或 [N]

### 真实证据复核

production smoke、hook log、session JSONL 能证明“真实发生过”，但仍需回到当前代码确认是否已修复。

**检查清单**：
- 记录证据来源路径和 session_id / timestamp
- 读取当前代码和测试，确认该问题仍可达还是已经被 regression 覆盖
- 若问题只存在于旧版本日志，报告为已修复记录或文档更新，不升级为 P0/P1
- 若当前代码没有测试覆盖，按影响分级为测试缺口，而不是直接判定功能失败

### 验证输出

| 标记 | 含义 | 行动 |
|------|------|------|
| ✅ 确认 | 真实问题 | 纳入 Phase 3 报告 |
| ⚠️ 部分正确 | 有意设计或影响极低 | 记录但不修复 |
| ❌ 误报 | 审计错误 | 排除，记录原因 |

## Phase 3：报告模板

```markdown
# ticketNN: PACEflow 全面审查

## 审查摘要

审查时间: YYYY-MM-DDTHH:mm:ss+08:00
审查范围: N 文件（动态发现：M hook + K skill + J 模板 + L 工具/文档）
证据基线: 当前 git HEAD + 工作区 diff + 动态发现文件；文档仅作候选线索

Phase 1 原始发现: NC + NH + NW + NI = Total
Phase 2 验证结果: X 确认 / Y 部分正确 / Z 误报
误报率: Z/Total (NN%)

确认发现统计: NC_confirmed + NH_confirmed + NW_filtered + NI_filtered

## 确认发现

### P0 必修
- [C-N] 文件名:行号 — 问题描述 → 建议修复

### P1 建议
- [H-N] / [W-N] 文件名:行号 — 问题描述 → 建议修复

### P2 文档
- [W-N] 文件名:行号 — 问题描述 → 建议修复

## 部分正确/有意设计
（记录为什么不修复）

## 误报分析
（记录误报原因，改进未来审查）

## 验证矩阵
（Phase 2 每个 C/H 发现的验证方法和结论）

```

## 修复优先级

| 优先级 | 条件 | 行动 |
|--------|------|------|
| **P0 必修** | 确认的 C 级 + 影响可靠性的 H 级 | 本次变更修复 |
| **P1 建议** | 确认的 W 级中代码质量问题 | 建议修复 |
| **P2 文档** | 文档过时、版本号不一致 | 顺手修复 |
| **P3 延后** | I 级优化建议 | 派 `artifact-writer record-finding` 记录到 `changes/findings/` |

## 误报防御（经验教训）

> 基于多轮审查经验总结的四大误报根因。验证 agent 必须注意这些陷阱。

### 1. 模式匹配非路径追踪

**症状**：看到变量名不一致就报 bug，未沿控制流验证是否可达。
**防御**：验证时必须从"问题行"追踪到入口点，确认执行路径。

### 2. 缺设计意图上下文

**症状**：把有意设计当 bug 报告。
**防御**：检查代码注释、测试名、hook 日志和必要的 CLAUDE.md 背景。明确标注设计意图的降级为 ⚠️。

### 3. 未实际 diff

**症状**：声称"模板不一致"或"文件不同步"，实际逐行对比完全一致。
**防御**：对任何"不一致"声称，必须真正读取并 diff 两个文件。

### 4. 严重度膨胀

**症状**：W 级问题升级为 C 级。
**防御**：C 级必须证明"功能错误或数据丢失"的具体触发路径，不能仅凭代码气味。

### 5. 文档驱动误判

**症状**：把 guidebook/action-plan 的历史缺口当成当前 bug，或把 README 的数量描述当成发布面事实。
**防御**：先用 Glob/rg 发现当前文件，再读 `.claude-plugin/**`、`hooks/hooks.json`、`hooks/pace-utils.js`、测试和真实日志。文档与代码不一致时，优先报告“文档过时”或“需要代码证据补充”。
