# 对抗审计方法论（通用版）

> 本文件是 PACEflow R（审计）阶段供主 session direct review subagent 的**通用对抗审计方法论**，任意项目、任意技术栈可用，不绑定任何特定代码库的目录结构、维度划分或版本基线。
>
> **定位（铁律）**：PACEflow 是流程执行纪律工具，不是质量控制工具。本方法论也只服务一个目的——让"对抗审计这一步真发生并被记录"，**从不裁决代码质量**。审计输出的是 P0-P3 分级 findings 报告，由主 session 按既有流程处置（开 HOTFIX 修、或 `record-finding` 记 backlog、或记 won't-fix）。审计**不卡 close**：只要审计跑过并记录处置，CHG 即可闭合，无论挖出什么。这里全部七条都是**发现纪律**，没有一条是"质量标准"。

---

## 方法论内核（七条）

### 1. 独立发现，不照文档打勾

审计的核心价值是**独立挖出问题**，不是对着 README / guidebook / 设计文档 / plan 逐条核对"做没做到"。

- 把改动当作**待证伪的对象**，主动找它哪里会坏，而不是确认它符合描述。
- 文档说"应该这样"不等于代码"真的这样"。审计要看代码实际行为，不替设计文档背书。
- 一个改动"看起来合理""通过了测试"都不是没问题的证据——审计的任务正是去找自动化没覆盖、设计意图与实现之间的漂移、对抗式边界回归这些"看起来对"底下的坑。

### 2. 证据优先级：代码行为是 ground truth

判定一个 finding 真假、定它的严重度时，证据按以下优先级采信：

```
代码 / 配置  >  测试 / fixture  >  真实运行日志 / session 记录  >  文档
```

- **代码与配置**是 ground truth：实际控制流、实际正则、实际分支、实际 I/O 协议决定行为。
- **测试与 fixture**次之：它们固化了预期契约，能证明"某行为被覆盖/未覆盖"。
- **真实运行日志 / session 记录**能证明"某问题真实发生过"，但仍需回到当前代码确认是否已修复（旧日志里的问题可能早被后续 commit 覆盖）。
- **文档级别最低**：README / 设计文档 / 注释**只能用来发现候选矛盾或理解设计意图，不能单独作为 C/P0 级 bug 的证据**。文档与代码不一致时，默认先采信代码行为，再把文档列为"可能过时"。任何 C/P0 级问题必须从代码路径、配置、测试缺口或真实日志里**独立证明**，不能仅凭"文档说会出事"就定级。

### 3. 报告全部 → 再验证（验证才是核心）

审计分两段，**绝不在第一段预筛**：

- **Phase 1 报告全部**：把所有可疑发现都报出来，严重度只做标注，**不因为"看起来低优先级""可能是文档问题"提前丢弃**。漏报比误报更危险——没报出来的就进不了验证。
- **Phase 2 验证去重降级剔误报**：对（尤其 C/P0 级）发现逐个验证，去重、降级、剔除误报，三分类 ✅ 确认 / ⚠️ 部分正确（有意设计或影响极低）/ ❌ 误报。

**历史经验：第一段原始发现的误报率高达 50-80%。所以验证（Phase 2）才是整个审计的核心，不是 Phase 1 的罗列。** 一份只有 Phase 1 罗列、没有 Phase 2 验证的报告基本不可用。

### 4. 三件核查武器

Phase 2 验证一个发现真假，靠这三件武器（按发现类型选用）：

| 武器 | 适用 | 做法 |
|------|------|------|
| **路径追踪** | 逻辑错误类发现 | 从"问题行"沿控制流追踪到入口点，确认这条路径**真的可达**、变量值在路径上满足触发前置条件。可达才算真，不可达即误报。 |
| **实际 diff** | "不一致 / 不同步"类声称 | 用工具**真正读取并逐行对比**两处实际内容，不能凭记忆断言"它们不一致"。 |
| **设计意图查证** | 疑似有意设计 | 读问题附近代码注释、测试名、fixture、就近文档，确认不是有意为之。明确标注为有意设计的，降级为 ⚠️ 而非 bug。 |

（可补充第四件：**最小复现**——能构造触发条件时，写一个最小测试/命令实跑一遍，是最硬的证据。）

### 5. 严重度纪律（反 severity 膨胀）

- **C / P0 级必须证明具体触发路径**：要能指出"在什么输入 / 什么状态下，沿哪条路径，触发什么错误结果（功能错误 / 数据丢失 / 流程阻塞）"。给不出具体触发路径的，不许定 C/P0。
- 严重度只升不降是反模式。"代码气味""不够优雅""可能有风险"不是 C/P0，按实际影响如实降级为 W/I。
- 反向也成立：**放宽 / 修复一个问题后，要主动验对称回归**——修 under-block 时 over-block 是对称风险，加宽匹配后要反向验有没有误伤。TDD 绿灯只证目标达成，不证没引入新误伤。

### 6. 误报防御七条

Phase 2 验证时，对照这七类高频误报根因逐一排雷：

1. **模式匹配非路径追踪**——看到变量名/字符串不一致就报 bug，没沿控制流验证是否可达。防御：必须从问题行追踪到入口。
2. **缺设计意图**——把有意设计当 bug。防御：先读注释 / 测试名 / 就近文档，确认非故意。
3. **未实际 diff**——声称"不一致 / 不同步"，实际逐行对比完全一致。防御：任何"不一致"声称必须真读两处并对比。
4. **严重度膨胀**——W/I 级硬升成 C/P0。防御：C/P0 必须给出具体触发路径，否则降级。
5. **文档驱动误判**——把设计文档 / plan 里的历史缺口或旧描述当成当前 bug，或把文档的数量/结构描述当成现状事实。防御：先用搜索发现当前真实文件与代码，再判定；文档与代码不符时优先报"文档过时"。
6. **过早过滤**——Phase 1 只报高严重度，真实但低严重度的发现被提前丢掉，Phase 2 无从复核。防御：Phase 1 全报，过滤是 Phase 2 的职责。
7. **大 diff stall**——对超长文件 / 大区间直接跑无界 diff，卡住最后没产出。防御：先用 `git log --stat <range> -- <file>` 定位相关 commit，再 `git show <sha> -- <file>` 逐个看；长文件先按函数名/标题用搜索定位小范围再读。

### 7. 记录审查基线（让审计可复现）

报告必须记录审查时的**基线快照**，否则结论无法复现也无法复核：

- **git HEAD**：审查时的 commit SHA。
- **工作区 diff 状态**：审的是已提交内容还是工作区未提交改动（本 CHG 场景通常是后者）；列出本次审计覆盖的改动范围。
- **证据来源**：列出实际读过/对比过/跑过的关键源码、配置、测试、日志、session 路径。**不要只列设计文档**——证据来源若全是文档，说明这份审计没落到代码 ground truth。
- **未覆盖范围**：因日志缺失、无法复现、时间或工具限制而未验证的部分，如实写明，不假装全覆盖。

## 如何用：主 session 编排一次 R 阶段审计

R 阶段是**主 session 的编排活**（派 subagent、读报告、判断 findings 处置），**不是 artifact-writer 的动作**——artifact-writer 只在收尾时落 REVIEWED 证据行。完整 close 流位置见 `../SKILL.md` 的 R（审计）步骤。

主 session 按本 CHG 的 diff 编排审计：

1. **看本 CHG 改了什么**：先过一遍本次 CHG 的实际 diff（改了哪些文件、哪类逻辑），作为选 review 棱镜的依据。

2. **按内容自选 review agent / 棱镜，不固化标准 agent**：根据改动内容选合适的审查视角，而非套用一套固定的"标准审查 agent 清单"。
   - 改了控制流 / 边界 / 正则 / 状态机 → 选偏**逻辑正确性 + 路径追踪**的棱镜；
   - 改了多处需要对齐的文件（接口与调用方、模板与解析方、文档与实现）→ 选偏**一致性 / 实际 diff**的棱镜；
   - 改了对外契约 / 配置 / 安全相关 → 选偏**协议合规 + 鲁棒性**的棱镜；
   - 琐碎、低风险的小改动 → 主 session 自己**瞄一眼**（`review-source: manual`）即可，不必派独立 agent。
   - **为什么自选**：固化一套"标准 agent"等于试图穷举所有场景 = 把隐性质量标准塞进流程、且脆。自选锁住"审计发生"、放开"用什么棱镜"。

3. **用本方法论 direct 所派的 subagent**：把上面**七条内核**作为审查纪律注入 subagent 的 prompt（独立发现 / 证据优先级 / 报告全部再验证 / 三件武器 / 严重度纪律 / 误报防御七条 / 记录基线），并明确要求它**输出 P0-P3 分级报告**：
   - **P0**：确认的 C 级——有具体触发路径的功能错误 / 数据丢失 / 流程阻塞；
   - **P1**：确认的高影响问题——影响可靠性但不直接阻塞；
   - **P2**：代码质量 / 文档过时类；
   - **P3**：优化建议 / 风格改进。
   - 报告须含：每条 finding 的 `文件:行号 + 触发路径或证据 + 影响 + 建议`，以及第 7 条要求的审查基线与证据来源。

   > **执行约束（硬性）**：审计 subagent 必须 **inline / foreground 派发**（Task / Agent 工具同步等待），**不可作为 background / detached 任务**。理由：inline 派发时主 session 阻塞在 tool call 上（mid-turn），审计必然在本轮收尾之前返回；若错误地派成 background，主 session 可能在审计在途时就 end-turn，撞上 Stop 的"未审计"拦截。

4. **读 P0-P3 报告并路由 findings**（主 session 判断，调既有 writer 操作）：
   - **P0 / P1** → 开 HOTFIX（`create-chg --type hotfix`）修，或判定不修则记 won't-fix finding（`record-finding`）；
   - **P2 / P3** → 派 `record-finding` 进 backlog；
   - **迭代闸**：审计-findings 生出的 HOTFIX **默认不自动重审**（深度=1），防止"审计→修→再审→再修"无止境递归。

5. **收尾写 REVIEWED**：findings 路由完成后，派 `close-chg`（含 `review-confirmed: true` / `review-source` / `review-findings`）一把梭折叠 VERIFIED + REVIEWED + 归档；只想记录审计暂不归档时，才派 `update-chg action=review`。字段格式见 Skill(paceflow:artifact-management)。

**再次强调**：审计是**阻断-on-步骤、不阻断-on-结论**——close 前必须"审计这步跑过并记录处置"，但绝不要求"P0/P1 修完"才让 close。一个审计挖出 5 个 P0、全部路由成 won't-fix，照样满足 REVIEWED。流程从不裁决质量，只确认审计发生且记录在案。
