# 审计流程详情

> 本文件是 paceflow-audit 的 Phase 2/3 详细流程和误报防御策略。

---

## Phase 2：验证方法详解

### 路径追踪

从报告的"问题行"出发，沿控制流追踪所有可达路径，确认是否真的会触发问题。

**检查清单**：
- 入口函数是否被调用？
- 条件分支是否可达？
- 变量值在执行路径上是否符合前置条件？

### 实际 diff

声称"不一致"的发现，逐行对比两个文件的实际内容。

**检查清单**：
- 用 Read 工具分别读取两个文件
- 逐行对比关键段落
- 确认差异是真实的还是审计 agent 的记忆偏差

### 设计意图查证

**检查清单**：
- 读取 CLAUDE.md "开发约定"章节
- 读取代码中相关注释
- 如果有注释说明"这是有意设计"，降级为 ⚠️

### 验证输出

| 标记 | 含义 | 行动 |
|------|------|------|
| ✅ 确认 | 真实问题 | 纳入 Phase 3 报告 |
| ⚠️ 部分正确 | 有意设计或影响极低 | 记录但不修复 |
| ❌ 误报 | 审计错误 | 排除，记录原因 |

## 修复优先级

| 优先级 | 条件 | 行动 |
|--------|------|------|
| **P0 必修** | 确认的 C 级 + 影响可靠性的 H 级 | 本次变更修复 |
| **P1 建议** | 确认的 W 级中代码质量问题 | 建议修复 |
| **P2 文档** | 文档过时、版本号不一致 | 顺手修复 |
| **P3 延后** | I 级优化建议 | 记录到 findings.md |

## 误报防御（经验教训）

> 基于多轮审查经验总结的四大误报根因。验证 agent 必须注意这些陷阱。

### 1. 模式匹配非路径追踪

**症状**：看到变量名不一致就报 bug，未沿控制流验证是否可达。
**防御**：验证时必须从"问题行"追踪到入口点，确认执行路径。

### 2. 缺设计意图上下文

**症状**：把有意设计当 bug 报告。
**防御**：检查 CLAUDE.md 开发约定和代码注释。明确标注设计意图的降级为 ⚠️。

### 3. 未实际 diff

**症状**：声称"模板不一致"或"文件不同步"，实际逐行对比完全一致。
**防御**：对任何"不一致"声称，必须真正读取并 diff 两个文件。

### 4. 严重度膨胀

**症状**：W 级问题升级为 C 级。
**防御**：C 级必须证明"功能错误或数据丢失"的具体触发路径，不能仅凭代码气味。
