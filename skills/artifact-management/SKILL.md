# Artifact 文件管理规则

管理核心 Artifact 文件的创建、更新和维护。

> **存储位置**：所有 Artifact 文件存储在 Obsidian Vault（`VAULT_PATH/projects/<projectName>/`）。`getArtifactDir(cwd)` 为唯一路径解析器，PreToolUse hook 自动将 CWD 路径重定向到 vault 路径。`.pace/` 运行时状态保留在项目 CWD。

---

## 核心 Artifact 文件

| 文件 | 用途 | 更新方式 | 技术限制 |
|------|------|----------|----------|
| `spec.md` | 项目元数据与技术栈 | 直接修改相关段落 | 使用 `Edit` 工具，无 ARCHIVE 标记 |
| `task.md` | 任务分解与进度追踪 | 活跃区修改状态，完成后归档 | **禁止 `Write` 覆盖**，使用 `Edit` 工具 |
| `implementation_plan.md` | 技术实现方案 | 活跃区更新，完成变更归档 | **禁止 `Write` 覆盖** |
| `walkthrough.md` | 工作总结与验证记录 | 活跃区更新最近记录，旧记录归档 | **禁止 `Write` 覆盖** |
| `findings.md` | 外部调研与发现记录 | 活跃区更新摘要索引，详情归档 | **禁止 `Write` 覆盖** |

**spec.md 同步触发词**：
以下操作触发 `spec.md` 更新：
- 安装新依赖（pip install / npm install / pnpm install）
- 添加新配置项（config.py / .env）
- 创建新核心模块（api/*.py / services/*.py）
- 框架/库版本升级

## 强制约束

### 创建文件

文件**不存在**时：使用 `Write` 工具创建

### 更新文件

文件**已存在**时：

- **spec.md**：使用 `Edit` 工具直接修改相关段落（无 ARCHIVE 标记）
- **task.md**：使用 `Edit` 工具在活跃区修改任务状态，完成后归档到 `<!-- ARCHIVE -->` 下方
- **其他三个文件**：使用 `Edit` 工具在 `<!-- ARCHIVE -->` 标记**上方**的活跃区更新

### 双区结构规则

除 `spec.md` 外，其他 4 个 Artifact 文件使用 `<!-- ARCHIVE -->` 标记分为活跃区和归档区：

```
┌─ 活跃区（ARCHIVE 上方）─┐  ← 当前状态、未完成项、最新内容
│  hook 自动读取此区域     │
├─ <!-- ARCHIVE --> ──────┤  ← 分隔标记（模板必须包含）
├─ 归档区（ARCHIVE 下方）─┤  ← 已完成的历史记录
└─────────────────────────┘
```

**写入规则**：
- 新内容写入活跃区（ARCHIVE 标记上方）
- 已完成项从活跃区移至归档区（ARCHIVE 标记下方）

**排列顺序**（统一倒序：新→旧）：
- **活跃区索引/条目**：最新的在最上方（新增条目插入索引顶部）
- **活跃区详情段落**：最新详情在上，旧详情在下
- **归档区**：最近归档的在 `<!-- ARCHIVE -->` 下方最前

**归档操作必须原子化**：使用单次 `Edit` 操作完成，`old_string` 跨越被归档内容和 `<!-- ARCHIVE -->` 标记，`new_string` 把内容移到标记之后。**禁止拆分为两次 Edit**（先删除再插入），中间失败会丢数据。

**task.md 归档时机**：
- 任务标记为 `[x]` 或 `[-]` 后，应归档到 ARCHIVE 下方
- PostToolUse hook 会检测活跃区中的已完成项并提醒归档

### 禁止事项

**严禁**使用 `Write` 工具覆盖以下文件：
- `task.md`（已存在时）
- `implementation_plan.md`
- `walkthrough.md`
- `findings.md`

**违规处理**：若发现覆盖更新导致历史记录丢失，必须立即恢复丢失内容。

**例外情况**：
- **文件损坏修复**：经用户明确确认后，允许使用 `Write` 工具重建损坏的 Artifact。

> [!IMPORTANT]
> Artifact 文件约束的权威定义在 **User Rule G-7**，本 Skill 遵循 G-7 规则。

## 状态标记规范

### 任务状态 (Task Status)

| 标记 | 含义 | 使用场景 |
|------|------|----------|
| `[ ]` | 未开始 | 任务已创建，尚未开始 |
| `[/]` | 进行中 | 正在执行的任务 |
| `[x]` | 完成 | 任务完成并验证 |
| `[!]` | 阻塞 | 遇到问题，需要帮助或等待外部依赖 |
| `[-]` | 跳过 | 任务取消或不再需要 |

**状态转换**：
- `[ ]` → `[/]` → `[x]`
- `[/]` ↔ `[!]`
- `[ ]` 或 `[/]` → `[-]`

### 并行任务标记 `[P]`

在任务描述中添加 `[P]` 标记，表示该任务可与同批次其他 `[P]` 任务并行执行：
```
- [ ] T-001 [P] 语法验证
- [ ] T-002 [P] diff 校验
- [ ] T-003 文档更新（依赖 T-001/T-002，无 [P]）
```
- 未标 `[P]` 的任务按顺序执行
- `[P]` 是纯约定标记，无 hook 检测

**更新时机**：
- 开始任务时：`[ ]` → `[/]`
- 完成任务时：`[/]` → `[x]`
- 遇到阻塞时：`[/]` → `[!]`（必须说明阻塞原因）
- 阻塞解除时：`[!]` → `[/]`
- 取消任务时：必须说明取消原因

### 变更状态 (Change Status)

```
[ ] 规划中 ─┬─→ [/] 进行中 ─┬─→ [x] 完成
            │               │
            └─→ [-] 废弃    └─→ [!] 暂停 ─→ [/] 进行中
```

| 状态 | 含义 | 触发条件 |
|------|------|----------|
| `[ ]` 规划中 | 方案已创建，待确认 | PACE 的 A 阶段完成 |
| `[/]` 进行中 | 用户已批准，正在实施 | PACE 的 C 阶段用户确认 |
| `[x]` 完成 | 所有任务完成并验证 | PACE 的 E 阶段完成 |
| `[-]` 废弃 | 需求取消或方案作废 | 用户明确取消 |
| `[!]` 暂停 | 阻塞或等待外部依赖 | 用户要求暂停 |

### 调研状态 (Findings Status)

> **注意**：findings.md 的状态标记与上述任务/变更状态是**独立的含义系统**，不可混用。

| 标记 | 含义 | 使用场景 |
|------|------|----------|
| `[x]` | 已采纳/已验证 | 调研结论确定，已应用或创建关联 CHG |
| `[-]` | 保持现状/已否定 | 调研否定该方案，不采纳（须注明理由） |
| `[ ]` | 参考中/待评估 | 调研尚未完成，待后续 session 评估 |

**流转规则**：结论确定 → `[x]`；创建关联 CHG → 补 `[change:: CHG-ID]`；否定 → `[-]` 并注明理由

## 任务编号规范

| 格式 | 说明 | 示例 |
|------|------|------|
| `T-NNN` | 三位数序号，全局递增 | T-001, T-042, T-100 |

**生成规则**：
1. 读取 `task.md` 中现有的最大任务编号
2. 新编号 = 最大编号 + 1（补零至三位）
3. 若无现有任务，从 T-001 开始
**使用场景**：
- `task.md` 中的任务清单项
- `implementation_plan.md` 索引表的「关联任务」字段
- `walkthrough.md` 完成记录中的任务引用

---

## 变更 ID 规范

| 格式 | 说明 | 示例 |
|------|------|------|
| `CHG-YYYYMMDD-NN` | 常规变更，日期+当日序号 | CHG-20260305-01 |
| `HOTFIX-YYYYMMDD-NN` | 紧急修复 | HOTFIX-20260304-01 |

**生成规则**：读取 `implementation_plan.md` 变更索引中当日最大序号，+1 生成新 ID。详细的变更管理流程参见 `paceflow:change-management` skill。

**注意**：任务编号 (T-NNN) 和变更 ID (CHG-...) 是不同的编号体系。T-NNN 标识单个任务，CHG-ID 标识一组关联任务的变更。

## 模板文件

创建新 Artifact 时，hooks 自动使用以下模板（位于 `hooks/templates/`）：

- `spec.md` — 项目规格模板
- `task.md` — 任务清单模板
- `implementation_plan.md` — 实施计划模板
- `walkthrough.md` — 工作记录模板
- `findings.md` — 调研记录模板

## 格式速查

> DENY/Stop/HINT 消息引用此处。Hook 检测正则为行首 Markdown checkbox 格式，表格或 emoji 格式无法识别。

### task.md 完整格式

```markdown
### CHG-YYYYMMDD-NN: 变更标题

<!-- APPROVED -->
<!-- VERIFIED -->

- [/] T-001 任务描述
- [ ] T-002 任务描述
- [x] T-003 已完成任务
```

- **CHG 分组**：每个变更独立 `###` 标题
- **`<!-- APPROVED -->`**：C 阶段获批后添加，放在 CHG 标题下方、任务列表上方
- **`<!-- VERIFIED -->`**：V 阶段验证通过后添加，放在 APPROVED 下方
- **任务条目**：`- [状态] T-NNN 任务标题`，状态为 `[ ]`/`[/]`/`[x]`/`[!]`/`[-]`
- **归档**：`[x]`/`[-]` 项用单次 Edit 移到 `<!-- ARCHIVE -->` 下方

### implementation_plan.md 完整格式

**索引条目**（活跃区顶部）：
```markdown
- [/] CHG-20260308-01 功能名称 #change [tasks:: T-001~T-003]
```

**详情段落**（活跃变更详情区）：
```markdown
### CHG-20260308-01 功能名称

描述变更内容和背景。

**T-001 任务标题**：
- 具体改动说明

**T-002 任务标题**：
- 具体改动说明
```

- 每个索引条目**必须**有对应的 `### CHG-ID` 详情段落
- Hook 检测 E 阶段前提：`/^- \[\/\]/m`（行首 `- [/] `）
- **禁止**：表格格式 `| 状态 | CHG-ID |` 或 emoji `✅`/`❌`

### 常见错误

| 错误格式 | 正确格式 | 说明 |
|---------|---------|------|
| `\| [/] \| CHG-... \|` | `- [/] CHG-...` | 表格 → checkbox |
| `✅ CHG-...` | `- [x] CHG-...` | emoji → checkbox |
| 索引无详情段落 | `### CHG-ID` 段落 | hook 守门 DENY |
| 双 `<!-- ARCHIVE -->` | 仅保留 1 个 | readActive 截断 |
