## 4. VERIFIED 完整规范（C9 主体）

### 4.1 双表示、单权威

| 维度 | 内容 | 用途 |
|------|------|------|
| 机器权威（单源） | frontmatter `verified-date: <ISO 8601 datetime>` | 状态时间戳；机械验证；hooks / agent 解析 |
| 人读 / hook 信号 | `<!-- VERIFIED -->` HTML 注释 | 视觉标记；hook 廉价文本检查 |
| 一致性约束 | 两者同时存在 ↔ 同时不存在 | 不一致即 `format-violation` |

### 4.2 frontmatter 字段

新增字段，与 `completed-date` / `archived-date` 同组：

```yaml
chg-id: CHG-YYYYMMDD-NN
status: completed | archived
date: YYYY-MM-DD
type: change | hotfix | research
parent-tasks: ["[[task]]"]
parent-impl: ["[[implementation_plan]]"]
related-finding: null
aliases: []
tags: []
schema-version: "6.0"
completed-date: <ISO 8601 datetime> | null
verified-date: <ISO 8601 datetime> | null   # 新增
archived-date: <ISO 8601 datetime> | null
```

datetime 格式强制：`YYYY-MM-DDTHH:mm:ss+08:00`（含日期 + 时间 + 时区，如 `2026-05-04T14:30:00+08:00`）。生成命令：`date '+%Y-%m-%dT%H:%M:%S+08:00'`。

### 4.3 详情文件正文位置

固定为 `<!-- APPROVED -->` 紧接下一行：

```markdown
- [x] T-901 任务一
- [x] T-902 任务二

<!-- APPROVED -->
<!-- VERIFIED -->

## 实施详情
```

**约束**：

- `<!-- VERIFIED -->` 必须紧跟 `<!-- APPROVED -->` 下一行（**不空行间隔**）
- 缺 `<!-- APPROVED -->` 而出现 `<!-- VERIFIED -->` → `format-violation`
- 上方至少留一空行（与任务清单结尾隔开）；下方留一空行（与 `## 实施详情` 隔开）

### 4.4 状态机集成

| frontmatter status | verified-date | `<!-- VERIFIED -->` | 索引 checkbox | 解释 |
|---|---|---|---|---|
| `planned` / `in-progress` | null | 缺 | `[ ]` / `[/]` | 进行中，未到 V 阶段 |
| `completed` | null | 缺 | `[x]`（活跃区） | 任务完成但 V 未通过 — block 状态 |
| `completed` | 有 | 有 | `[x]`（活跃区） | V 通过，待归档 |
| `archived` | 有 | 有 | `[x]`（ARCHIVE 下方） | 已归档 |
| `archived` | 有 | 缺 | `[x]`（ARCHIVE 下方） | `format-violation` |
| `archived` | 缺 | 有 | `[x]`（ARCHIVE 下方） | `format-violation` |
| `cancelled` | null | 缺 | `[-]`（ARCHIVE 下方） | 取消，不验证 |

### 4.5 写入路径

唯一允许：派 `paceflow-artifact-writer` 执行 `update-chg action=verify`（详见 §5）。

主 session 直接 Edit `<!-- VERIFIED -->` 或 `verified-date` → 流程违规（hook 应在 P0.5 拦截，本 spec §10.2 已详记期望）。

## 5. update-chg action=verify 完整规范（C10 主体）

### 5.1 输入字段

```yaml
operation: update-chg
fields:
  target: "CHG-YYYYMMDD-NN"      # 必填
  action: verify                  # 必填
  verify-summary: "..."           # 选填，写入 ## 工作记录 单元格
```

不需要 `section`（verify 是元操作，不写 section）；不需要 `task-id` / `new-status`（status 由 update-status 负责，verify 不动 status）。

### 5.2 操作子流程

1. Read `changes/<id>.md`（target 解析失败 → `target-not-found`）
2. **前置校验**（任一失败即 `format-violation`，不写文件）：
   - frontmatter `status` 必须等于 `completed`
   - `<!-- APPROVED -->` 必须存在
3. **幂等检查**（已完整验证 → SUCCESS 幂等，不写文件）：
   - frontmatter `verified-date` 已有非 null 值 **AND** `<!-- VERIFIED -->` 已存在
   - 报告 `status: SUCCESS`，`reason: "already verified, no change"`
4. **不一致检查**（任一失败即 `format-violation`，不写文件）：
   - `verified-date` 已有但 `<!-- VERIFIED -->` 缺失
   - `<!-- VERIFIED -->` 存在但 `verified-date` 为 null
5. **写入**（按顺序，原子语义）：
   - Edit frontmatter：`verified-date: <date '+%Y-%m-%dT%H:%M:%S+08:00' 输出>`
   - Edit 详情正文：在 `<!-- APPROVED -->` 行之后、紧邻插入 `<!-- VERIFIED -->`
   - Edit `## 工作记录` 表格末尾追加：`| <YYYY-MM-DD> | 验证通过：<verify-summary 或 "无附加说明"> |`
6. 报告 `status: SUCCESS`，`files_modified: ["changes/<id>.md"]`

### 5.3 边界条款

| 触发条件 | 失败码 |
|---------|-------|
| target CHG 文件不存在 | `target-not-found` |
| frontmatter `chg-id` 与文件名不匹配 | `id-mismatch` |
| status ≠ `completed` | `format-violation` |
| 缺 `<!-- APPROVED -->` | `format-violation` |
| `## 工作记录` 表格缺失 | `format-violation` |
| 已完整验证（幂等） | **SUCCESS**（reason: already verified） |
| `<!-- VERIFIED -->` 与 `verified-date` 不一致 | `format-violation` |

### 5.4 update-chg.md 改动定位

**L10**（输入字段段）：

```diff
- `action`（必填，枚举：`append` | `replace` | `update-status` | `approve`）
+ `action`（必填，枚举：`append` | `replace` | `update-status` | `approve` | `verify`）
```

**L53 之后新增子流程章节**（在 `### action=approve` 之后插入 `### action=verify` 整段，~30 行）

**L93 边界区新增条款**（追加到现有边界列表末尾，6 条新边界）

## 10. Hook 期望（仅文档，留 P0.5 实施）

P0.5 hooks 改造时直接对照本节实施。

### 10.1 pre-tool-use.js（E 阶段守门）

代码文件 Edit 前要求：

- `task.md` / `implementation_plan.md` 至少一个 wikilink 索引行 `[/] [[chg-id]]`
- 对应 `changes/<chg-id>.md` 存在
- 详情文件必须含 `<!-- APPROVED -->`
- 详情 frontmatter `status: in-progress` **或** 任务清单含 `[/]`（已开始执行）

不依赖 `<!-- VERIFIED -->`（V 阶段才需要）。

### 10.2 stop.js（V 阶段验证）

会话即将结束时，对每个 `[x]` 索引：

- 对应详情 `status: completed` AND `verified-date: null` → block，提示"派 `update-chg action=verify`"
- 对应详情 `verified-date` 有值但 `<!-- VERIFIED -->` 缺失 → block，提示"标记不一致，派 `update-chg action=verify` 修复"
- 反之亦然（`<!-- VERIFIED -->` 存在但 `verified-date` 为 null）

### 10.3 post-tool-use.js（schema 校验）

Edit `changes/<id>.md` 后：

- 读 frontmatter `verified-date` 与正文 `<!-- VERIFIED -->` 计数
- 不一致 → 累积警告（提示派 agent 修复，不阻止 — 因为可能在中途）

### 10.4 archive-chg 前置（C12 实施）

`archive-chg.md` 已要求所有任务 `[x]/[-]`。P0.1 必须补加：

```
- frontmatter `status: completed` AND `verified-date` 不为 null AND `<!-- VERIFIED -->` 存在
- 任一缺失 → 报告 format-violation，提示派 update-chg action=verify
```

同时收紧现有前置：

- `status=in-progress` 不再允许归档，即使任务清单已经全 `[x]`；必须先由 `update-chg action=update-status` 推到 `status: completed`
- 活跃索引行应是 `[x] [[chg-id]]`，不是 `[/] [[chg-id]]`
- 归档移动时从活跃区删除 `[x]` 行，并在 `<!-- ARCHIVE -->` 下方插入同一 `[x]` 行

同时更新边界条款：

- `verified-date` 为 null → `format-violation: not verified`
- `<!-- VERIFIED -->` 缺失 → `format-violation: verified marker missing`
- `verified-date` 与 `<!-- VERIFIED -->` 不一致 → `format-violation: verification state inconsistent`
- `status=in-progress` 且任务全完成 → `format-violation: status not completed`
