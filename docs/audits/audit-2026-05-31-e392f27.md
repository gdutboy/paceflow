## 改动意图与设计评价

### 解决的问题

主 session 把工作委托给后台 Workflow / subagent / agent team / 后台 shell 后，需要 Stop 等待结果返回；但 running CHG 仍有未完成 T-NNN，旧 Stop 逻辑会硬阻断（exit 2），把"已委托后台执行"误判为"试图带未完成任务退出"。Claude Code v2.1.145+ 在 Stop 输入提供 `background_tasks`，本修复据此把"running CHG 未完成任务 + 存在活跃后台任务"降级为可见 `systemMessage` 软放行。

### 控制流嵌入（路径追踪）

```
for change of activeChanges:
  completionPending += change.tasks.pending        // 行210：先累加（在 running 分支之前）
  if change.category === 'running':
    if !approved: addWarning(...); continue          // 未批准照常硬阻断
    if change.tasks.pending > 0:
      if hasBackgroundTasks:                          // 行218：新增分支
        backgroundWorkReminders.push(...); continue   // 只发软提醒，不加 warnings[]
      addWarning('execution', ...); continue          // 无后台任务→照旧硬阻断
...
if warnings.length > 0: <硬阻断 exit 2>
else:
  if emitAllowedStopReminders({deferred, background, ...}): exit 0   // 行372：软放行
```

**设计正确性**（3 个关键不变量均保留）：

1. **虚假完成声明拦截不被绕过**：`completionPending` 在行210累加（早于 background continue），行283 `COMPLETION_PHRASES.test(lastMessage) && completionPending > 0` 独立于新分支生效——即便后台任务在跑，只要 AI 说了完成词且仍有 pending，照样 `addWarning('execution')` → 硬阻断。
2. **结构损坏 / 未验证 / 待归档硬阻断保留**：这些 reason 走各自的 `addWarning('repair'/'verify'/...)`，不进入新的 background 分支，warnings>0 时仍 exit 2。
3. **计数器重置**：软放行经 `emitAllowedStopReminders` 调 `resetHardBlockRuntime()`，正确清零 stop-block-count（避免遗留降级状态）。

`emitSoftReminders` 被重构为更通用的 `emitAllowedStopReminders`（同时处理 deferred 软提醒与 background 软放行），消除重复，命名清晰。

## 确认发现

### [E392-W1] background 状态过滤的终态黑名单是死逻辑（基于错误假设）（W，建议修）

- **位置**: `plugin/hooks/stop.js:64-66`（`activeBackgroundTasks` 的过滤正则）
- **描述**: `activeBackgroundTasks` 用终态黑名单判活跃——`return !/^(done|complete|completed|finished|success|succeeded|failed|error|cancelled|canceled|stopped)$/.test(status)`。该设计**隐含一个假设：已完成/失败的任务会带终态 status 留在 `background_tasks` 数组里，需要过滤掉**。**实测证伪了这个假设**：Claude Code v2.1.158 在任务完成/失败/结束时**直接把它移出数组**，从不以终态 status 保留。因此这 11 个终态词在真实数据里**永不出现**——黑名单过滤的是一组不存在的值，是死逻辑。
- **实测证据**（独立探针，12 次 Stop 快照）：
  - status 跨 4 形态（shell/subagent/workflow/teammate）**只观测到 `running`**
  - subagent 完成 → 从数组消失（捕获[5]）；shell 完成 → 消失（[6]）；**workflow 含 1 个 `exit 1` 失败子 agent 自然结束 → 数组归空 `[]`（[10]→[11]）**
  - 即便复合 workflow 内部有失败，`background_tasks` 整体项也不留 `failed`/`completed`，直接移除
- **次要风险（仍成立但概率极低）**：黑名单对 status 为空 `''` 或缺失的 task 项判为 active（`String('').trim()` 不匹配黑名单 → active）。即 `[{}]`/`[{status:''}]` 会触发软放行。但实测显示真实 task **总带 `status:"running"`**，故现实触发概率极低（需 Claude Code 传畸形项）。
- **影响**: 当前**无功能错误**——因为真实 task 都是 `running`，黑名单对 `running` 正确返回 active，软放行逻辑工作正常。问题是**代码表达了错误的领域模型**：维护者会以为"数组里有完成任务需过滤"，而真相是"数组里只有活跃任务"。死逻辑 + 误导性模型，故 W。
- **建议（基于实测）**: 简化为正向判断，让代码反映真实模型——
  ```js
  // 真实模型：background_tasks 里存在的项即活跃任务（完成即移除）
  function activeBackgroundTasks(input) {
    const raw = input && input.raw && typeof input.raw === 'object' ? input.raw : {};
    const tasks = Array.isArray(raw.background_tasks) ? raw.background_tasks : [];
    return tasks.filter(t => t && typeof t === 'object' && String(t.status || '').trim() !== '');
  }
  ```
  - 过滤非对象/空 status 项（堵住畸形项触发软放行的次要风险）
  - 删除终态黑名单（死逻辑）；若想防御未来 API 变化（万一某天 Claude Code 改为保留终态），可保留但**必须加注释**说明"当前 API 完成即移除，此黑名单是面向未来的防御，当前永不命中"
  - 注：此修法把 E392-I1 的死分支一并清掉（直接读 `raw.background_tasks`）
  - 补测试：空 status / 畸形 task 项不触发软放行；`status:"running"` 正常软放行

### [E392-W3] session_crons 字段未纳入"后台活动"判断（W，实测衍生）

- **位置**: `plugin/hooks/stop.js`（`activeBackgroundTasks` 只读 `background_tasks`，未读 `session_crons`）
- **描述**: 实测发现 Stop input 顶层除 `background_tasks` 外还有 `session_crons: []`（session 级 cron 任务数组）。`e392f27` 完全未处理它。如果 session 有活跃 cron 任务（定时重复执行），主 session 停下时它们仍会触发——但 PACEflow 不把 cron 视为"后台活动"，可能对"running CHG + 有 cron"场景仍硬阻断。
- **影响**: 未确认 cron 任务的实际语义（是否算"主 session 在等待的后台工作"）。若算，则是与 background_tasks 同类的遗漏；若不算（cron 是独立调度，不阻塞主流程），则当前行为正确。**属待确认的设计问题**，非确证缺陷，故 W 并标注未定。
- **建议**: 确认 `session_crons` 语义。若活跃 cron 应被视为"允许 Stop 等待"的信号，把它纳入 `activeBackgroundTasks` 或新增 `activeCrons` 判断；若不应，加注释说明有意忽略。

### [E392-I1] activeBackgroundTasks 第一分支恒假（死代码）（I）

- **位置**: `plugin/hooks/stop.js:60-62`
- **描述**: `const tasks = Array.isArray(input && input.background_tasks) ? input.background_tasks : (Array.isArray(raw.background_tasks) ? raw.background_tasks : [])`。这里 `input` 是 `parseStdinSync()` → `parseHookStdin()` 的返回值，其顶层字段是白名单式提取（`session.js:28-46`：sessionId/toolName/filePath/.../raw），**不含 `background_tasks`**。故 `input.background_tasks` 恒为 undefined，第一个 `Array.isArray(...)` 恒 false，实际永远走 `raw.background_tasks` 回退分支。
- **证据**: `session.js:28-46` parseHookStdin 返回对象列出全部顶层字段，无 background_tasks；`:45 raw: parsed` 才含原始 stdin 全字段。功能正确（回退分支生效，测试 10a1/10a2/10a3 通过），但第一分支不可达。与上一轮 X-1 / PM-1 同类"看似覆盖两路、实际只走一路"。
- **影响**: 无功能影响，纯冗余/误导性代码——维护者会误以为顶层 input 也可能携带 background_tasks。
- **建议**: 二选一——(A) 删除第一分支，直接 `const tasks = Array.isArray(raw.background_tasks) ? raw.background_tasks : []`；(B) 在 `parseHookStdin` 顶层正式提取 `backgroundTasks: parsed.background_tasks || []`（与其他白名单字段一致），使第一分支可达且语义统一。
