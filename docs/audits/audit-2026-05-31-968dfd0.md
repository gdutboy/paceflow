## P0/关键发现

### [F6-5] agent-tests 手写 YAML parser 误解析真实用例（H，regressed，三方坐实）

- **位置**: `tests/agent-tests/helpers/subagent-runner.js`（新增 `parseSimpleYaml` / `stripYamlComment`）
- **对应历史发现**: AUDIT-06-T2（框架硬依赖 python3+PyYAML，Windows 不可用）的修复**自身引入的回归**
- **描述**: 968dfd0 为消除 python3 依赖，手写了 YAML parser。但 `stripYamlComment` 被无差别 `.map()` 到**所有行**（`content.split('\n').map(stripYamlComment)`），发生在区分 block scalar 之前——导致 `body: |` block scalar 内部的 markdown 标题行（行首 `#`）被当 YAML 注释**整行清空**；同时 `parseScalar` 不处理行尾 `# 注释`，使标量值携带注释文本。
- **主 session 一手实测**（铁证）：用真实 `loadYaml` 解析 `tests/agent-tests/cases/phase-a/tc-a4-record-finding.yaml`——其 `body: |` 含 3 行 markdown 标题（`# 背景`/`# 发现`/`# 调研来源`，原文 :20/:24/:30）。解析后 `input.fields.body` 中这 3 个标题**连同文字全部丢失**（变为空行 `\n\n`），正文保留。判定 `s.includes('# 背景')`/`includes('背景')`/`includes('发现')`/`includes('调研来源')` **全部 false**。
- **影响范围**（subagent + 对抗验证）：5 个用例 body 中全部 markdown 标题被吃；6 个用例 12 处标量值携带行尾注释 → `max_tokens`（变 NaN）、`frontmatter_status`、`filename_match`、`files_created` 路径查找等比较永久误判 FAIL；tc-b5"过短拒绝"断言对象被破坏；production prompt 字段同样污染。
- **为何绿灯没发现**（与 F6-4 互证）：agent-tests 框架**完全独立于** `node tests/*` 主测试套件（实测 `grep subagent-runner|loadYaml tests/test-hooks-e2e.js tests/test-pace-utils.js` = 0 引用）。256/256 全绿与该框架无关。且 `run-tests.js dummy` 自测固定用 `tc-a1`（无 block scalar、无行尾注释），**正好绕开两条 buggy parser 路径** → 框架"自测通过"给出虚假信心。
- **对抗验证**: confirmed（failureMode=over-fix）。旧 python3/PyYAML 路径用 `safe_load` 按 YAML 规范正确处理（行尾注释剥离、block scalar 内 `#` 保留）——这些用例在 Linux/mac 原本解析正确（仅 Windows 崩溃）；新通用 parser 让**所有平台统一变成错误解析**。
- **建议**: 手写 parser 必须区分 block scalar 上下文——`stripYamlComment` 不能在解析前无差别 map，应在逐行解析时按当前是否在 block scalar 内决定是否剥离；`parseScalar` 需正确处理行尾 `#`（引号内除外）。或改用一个最小但符合 YAML 规范的依赖（纯 JS 的 yaml 库）。补：让 dummy 自测覆盖含 block scalar + 行尾注释的用例（堵自测盲区）。
