## 确认发现

### P0 必修（C / 高影响 H）

- **[H-1] `plugin/hooks/pace-utils/locks.js:274-276` + `plugin/hooks/pre-tool-use.js:833-857`** — Artifact 资源锁的 read path 无 TTL 自愈。
  - 触发路径: `readArtifactResourceLock()` 内部仅调 `readJsonLock()`（locks.js:181-203），不调用 `jsonLockIsStale()`（line 205-210）。`pre-tool-use.js:834` 直接 `if (existingLock.ok)` → `DENY_ARTIFACT_CONCURRENT_WRITE`。对比 `acquireJsonLock()`（line 212-249）的 EEXIST 分支有 `jsonLockIsStale` + `unlinkSync` 自愈，read path 完全没有。
  - 实际可达场景: artifact-writer 异常 kill（断网、SIGKILL、机器掉电），`SubagentStop` 没机会执行 → `.pace/locks/artifacts/detail:changes_chg-XXX.lock` 留为孤儿。下一次主 session/teammate agent 用 **Edit/MultiEdit** 编辑同一 artifact 时永久 deny（`pre-tool-use.js:880` 的 `DENY_DIRECT_ARTIFACT_WRITE` 只覆盖 `toolName === 'Write'`，不兜底 Edit）。
  - 影响: fail-closed 但不可自愈，单次故障污染后续所有 session；recovery 需要用户手动 `rm .pace/locks/artifacts/*.lock`。SessionStart/Stop/PostToolUseFailure 都不扫描孤儿。
  - 建议修复: 在 `readArtifactResourceLock` 内部或 `pre-tool-use.js:834` 调用前，对 `existingLock` 调 `jsonLockIsStale(lock, ARTIFACT_RESOURCE_LOCK_TTL_MS)`；为 true 时 `unlinkSync` 并视为 `ok: false`。或者 `SessionStart` 启动时 sweep `.pace/locks/artifacts/` 扫描所有 `*.lock`，stale 即删。

### I-19 取宽口径的语义与潜在风险

**审计建议（基于 audit 报告 I-19）**：在原 `parseChangeIndex` 的 embedded malformed 分支里加 `if (raw.match(/\[\[(chg|hotfix)-/i) && !matched) malformed.push(raw)`，即"含 wikilink 但不命中主正则的行加入 malformed"。

**实际修法（`586d4a8` `change-analysis.js`）**：直接删掉原来的 tag 过滤行：

```diff
       if (!embedded) continue;
-      if (!/(#(?:change|hotfix|research)\b|\[tasks::)/i.test(embedded[4] || '')) continue;
       m = embedded.slice(1);
       malformed = true;
```

原 `if` 的设计意图是「只有 embedded 行尾含 `#change/#hotfix/#research` tag 或 `[tasks::` dataview 字段时，才视为真正的索引行；否则当成正文段落里的引用跳过」。

**修改方向正确**：`parseChangeIndex` 只处理 `task.md` / `implementation_plan.md` 的活跃区文本，这两个文件结构化严格，仅含标题、`<!-- ARCHIVE -->` 标记与索引行——**理论上不会有正文段落引用** `- [ ] [[chg-...]]`。原 tag 过滤其实是过紧，会让"缺 #tag 但格式其他部分正确"的索引行被静默跳过，反而失去发现真正格式错误的机会。

**潜在风险（值得下次 audit 单独追踪）**：

1. 若未来在 `task.md` / `implementation_plan.md` 顶部或 `<!-- ARCHIVE -->` 内引入注释段、分组标题、说明段（如 `> 本组任务对应 [[chg-20260520-01|功能 A]]，预计本周完成`），只要这行命中 `malformedRe`（允许任意行首前缀），就会被标 malformed 并触发 hook warning。
2. 主观判断"哪些行算索引、哪些算说明"在新口径下完全交给正则：`^(.+)- [<x>] [[chg/hotfix-...]] ...`。如果维护者将来想加分组说明，可能因为 hook 报警而被劝退。

**建议长期方案**：

- 短期：依赖现有 `task.md` / `implementation_plan.md` 结构纪律（只放索引），用例上未出现误报即视为可接受。
- 中期（如出现误报）：恢复 tag 过滤，但条件改为「embedded 行**前缀**是 inline 文档标记（`>`、`<!--`、` `开头多空格的子项）」时跳过，避免把 doc 段落里偶然出现的 `- [ ] [[chg-...]]` 错杀。
- 测试覆盖：`tests/test-pace-utils.js` 已新增"无 tag malformed 粘连"用例锁住当前行为；若改回过滤需同步更新用例。

### 关于"PACE 自宿主下的 Bash 拦截"——澄清

在 Phase 2 验证以及 follow-up 复测时，主 session 跑 `node tests/test-pace-utils.js`（不含 redirect / pipe）被 `pre-tool-use.js` 拦截，stderr 文案为：

```
禁止使用 Bash 修改 artifact 文件。... 被拦截的命令：node tests/test-pace-utils.js
```

第一直觉是「PACE 自宿主 + Bash gate 偏严，把 read-only Bash 也拦了」。**这个解释不准确**，真实路径如下：

1. `pre-tool-use.js:610-614` 的 `mutatesArtifact` 判断是 OR，其中一项是 `bashCommandEmbedsArtifactWriteScript(bashCommand, cwd, artDir)`。
2. `bash-guard.js:318-337` `bashCommandEmbedsArtifactWriteScript` 会：
   - 解析命令的执行目标（`bashScriptExecutionTargets` → 提取 `tests/test-pace-utils.js` 作为脚本路径）；
   - `fsStat`/`fsRead` **静态读取脚本源码**；
   - 检查脚本内容是否同时含 ① 写 API（`fs.writeFileSync`、`fs.appendFileSync` 等）、② artifact 路径引用（`task.md`、`changes/`、`findings.md` 等模式）。
3. `tests/test-pace-utils.js` 作为单元测试脚本，**两条都命中**：里面通过 `fs.writeFileSync` 写 fixture artifact 文件（用来验证 hook 行为），且字符串里直接引用 `task.md` / `changes/` 等路径。
4. 因此 guard 把测试脚本判定为"嵌入式 artifact 写入脚本"，从源头 deny。

**这是 PACE 自宿主（dogfooding）的真实代价**，不是误拦：

- guard 没办法区分"测试时写隔离 fixture artifact"和"生产时写真实 artifact"——它只看脚本静态文本是否含 write API + artifact 路径模式。
- `node verify.js`（在仓根，不写任何 artifact 文件）不会触发该路径，所以 verify.js 能跑、测试脚本不能跑——差别在脚本源码内容，不在 pipe / redirect。
- 用户在 commit message 中跑测试的方式是直接在 shell 跑（绕过 Claude Code session 的 PreToolUse），那条 deny 路径只对**通过 Claude Code 主 session 的 Bash 工具**生效。

**若希望主 session 内也能跑这套测试**，可选方案：

- 让 guard 信任带特定标记的脚本（如 frontmatter `// pace-test-fixture: true` 注释、或 fixture 路径专门放 `tests/.tmp/`）；
- 在 hook 检测到执行目标是 `tests/test-*.js` 时跳过 `bashCommandEmbedsArtifactWriteScript`（但风险是 user 真的有恶意脚本就放 tests/ 下绕过）；
- 维持现状，承认 PACE 自宿主项目下"测试必须在 session 外跑"——目前 commit message 已用这条路径自验。

后两条是当前事实。若决定改第一条，应在 [I-20] 同区域加白名单逻辑并补回归测试。
