## 确认发现

### P0 必修（C / 高影响 H）

- **[H-1] `plugin/hooks/pace-utils/locks.js:274-276` + `plugin/hooks/pre-tool-use.js:833-857`** — Artifact 资源锁的 read path 无 TTL 自愈。
  - 触发路径: `readArtifactResourceLock()` 内部仅调 `readJsonLock()`（locks.js:181-203），不调用 `jsonLockIsStale()`（line 205-210）。`pre-tool-use.js:834` 直接 `if (existingLock.ok)` → `DENY_ARTIFACT_CONCURRENT_WRITE`。对比 `acquireJsonLock()`（line 212-249）的 EEXIST 分支有 `jsonLockIsStale` + `unlinkSync` 自愈，read path 完全没有。
  - 实际可达场景: artifact-writer 异常 kill（断网、SIGKILL、机器掉电），`SubagentStop` 没机会执行 → `.pace/locks/artifacts/detail:changes_chg-XXX.lock` 留为孤儿。下一次主 session/teammate agent 用 **Edit/MultiEdit** 编辑同一 artifact 时永久 deny（`pre-tool-use.js:880` 的 `DENY_DIRECT_ARTIFACT_WRITE` 只覆盖 `toolName === 'Write'`，不兜底 Edit）。
  - 影响: fail-closed 但不可自愈，单次故障污染后续所有 session；recovery 需要用户手动 `rm .pace/locks/artifacts/*.lock`。SessionStart/Stop/PostToolUseFailure 都不扫描孤儿。
  - 建议修复: 在 `readArtifactResourceLock` 内部或 `pre-tool-use.js:834` 调用前，对 `existingLock` 调 `jsonLockIsStale(lock, ARTIFACT_RESOURCE_LOCK_TTL_MS)`；为 true 时 `unlinkSync` 并视为 `ok: false`。或者 `SessionStart` 启动时 sweep `.pace/locks/artifacts/` 扫描所有 `*.lock`，stale 即删。
