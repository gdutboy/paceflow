## 确认发现

### P0 必修

#### [AUDIT-A01] bash-guard `commandTextLooksMutating` 段首锚点缺 `\n`——换行分隔写命令对 artifact fail-open（✅ confirmed P0/C，主 session 端到端实测 + 单元级双重坐实）

- **位置**: `plugin/hooks/pre-tool-use/bash-guard.js:88-93`（`commandTextLooksMutating`）；deny 组合点 `pre-tool-use.js:712-716`
- **描述**: `commandTextLooksMutating` 的每条主正则段首锚点为 `(^|[;&|]\s*)`，字符类**未包含 `\n`**。JS 正则无 `m` 标志时 `^` 只匹配整串起点。因此多行 Bash 命令里，凡位于非首行、且以**换行**（而非 `;&|`）与前文分隔的 mutating 动词（`rm`/`mv`/`cp`/`sed -i`/`truncate`/`chmod`/`dd`/`touch`/`git checkout`，`:88`；`find -delete`/`-exec`，`:89`）都不被识别为 mutating。这些动词无脚本引擎路径、无 write-api 路径，**唯一依赖** `commandTextLooksMutating`，故 `:715` 的 `bashCommandLooksMutating && referencesArtifact` 组合彻底失效，artifact 被静默改写/删除。
- **主 session 单元级实测**: `bashCommandLooksMutating('echo hi\nrm task.md')` = **false**；`'echo hi\nsed -i s/a/b/ task.md'` = false；`'echo ok\nmv task.md'` = false。对照 `'rm task.md'`（裸）/ `'echo hi; rm task.md'`（分号）/ `'echo hi | rm task.md'`（管道）全 = **true**。源码锚点对照：`:88-93` 锚点 `[;&|]` 全部 ❌缺 `\n`；`:102/:128/:132` 锚点 `[\n;&|]` ✅含 `\n`；`powershell-guard.js:126` 锚点 `[\n;|]` ✅含 `\n`。
- **主 session 端到端实测（铁证）**: 真实 `pre-tool-use.js` hook 子进程 + 隔离 v6 fixture（artifact 信号 + 活跃 running CHG），token 经 charcode 构造避免脚本自拦：

  | 命令 | decision | 判定 |
  |------|---------|------|
  | `rm task.md`（裸）| **deny** | ✅ 拦截 |
  | `echo hi; rm task.md`（分号）| **deny** | ✅ 拦截 |
  | `echo hi\nrm task.md`（**换行**）| **allow** | ❌ **fail-open，artifact 被删** |
  | `echo hi\nsed -i ... task.md`（换行）| **allow** | ❌ **fail-open，被改写** |
  | `echo ok\nmv task.md z.md`（换行）| **allow** | ❌ **fail-open** |
  | `echo ok\nrm implementation_plan.md`（换行）| **allow** | ❌ **fail-open** |
  | `cat task.md`（只读对照）| allow | ✅ 正确 |

- **修复建议**: `close-chg.md` 增守卫：若任务全为 `[-]`，报 `format-violation: all tasks skipped, use cancelled + archive-chg`，不写 completed。关联补 cancelled archive-only 正向成功路径回归测试。

## 系统性风险（本轮核心价值）

1. **bash↔powershell 双 guard 手工移植，反复多向不对称（最高频根因，连续多轮）**: 本轮一次暴露三个新方向——AUDIT-A01（bash 缺 powershell 的 `\n` 锚点，P0）、AUDIT-A02（powershell 缺 bash 的 script-engine 闸门，P1）、AUDIT-A08（powershell 缺 bash 的真实别名集 + git 还原拦截，P1）。加历史 AUDIT-01（powershell 缺 bash 的 H-1）。**强烈建议抽取共享 mutating 动词表/别名表/write-API 集/段首锚点/script-engine 闸门为单一模块，两 guard 共用。** 这是根治反复不对称的唯一办法。

2. **守卫启发式 altitude 不匹配**: `bash-guard.commandTextLooksMutating` 是扁平 "token 存在性 OR"（rm/mv/cp 出现在任意段即判 mutating），**无 source/target 路径解析**，而 redirect/named-write 路径**有**目标解析。这单一设计同时制造 AUDIT-A01（P0，漏 `\n`）和 D2-2（W，cp-source 误拦）。**建议把 mutating 路径收敛到 redirect 路径式的目标解析。**

3. **单点解析器污染整个 hook 矩阵**: AUDIT-A07 的大写 `[X]` bug 在 `change-analysis.js:143` 一处，却同时喂错全部 6 个运行时 hook。任何 change-analysis 的解析缺陷都会一致地放大到 session-start/stop/pre-tool-use 的判定。**建议 change-analysis 的 checkbox/status 解析补齐大小写/边界单测，作为全 hook 的共享地基。**

4. **守卫信号门控不对称（artifact 写保护的 Edit/Write 分裂）**: AUDIT-A04——同一类 artifact 写保护，Write 兜底对全信号生效、Edit 兜底只对 artifact 信号生效。**建议把所有 artifact 写保护兜底统一提到信号门控之外。**

5. **共享日志 1MB 截断 × e2e 字节 delta 断言（系统性 flaky）**: 见 AUDIT-W2，多维度独立指向，确属测试设计结构性脆弱。

6. **fail-open 单点全景**: 8 个 hook 共享 `require('./pace-utils')` 失败统一 fail-open；stop/pre-tool-use 外层 catch 行为需统一审视。pace-utils 任一子模块语法错误即令整个 PACE 守卫矩阵静默失效——发布前测试是唯一防线（已有 160/160 + 257/257 把关）。
