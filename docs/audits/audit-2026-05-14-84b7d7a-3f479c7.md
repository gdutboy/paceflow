### H（High） — 1 项

#### H-1（新引入）：`getNativePlanPath` 在 Windows 上吞掉路径大小写

- **位置**：`plugin/hooks/pace-utils/plans.js:129`
  ```js
  return ctx.normalizePath(path.resolve(cwd || process.cwd(), planPath));
  ```
- **根因**：`normalizePath` 在 win32 上做 `toLowerCase()`（`plugin/hooks/pace-utils/path-utils.js:34-37`），`getNativePlanPath` 直接把 normalize 结果返回给调用方
- **触发路径**：任何调用 `getNativePlanPath(cwd)` 的代码（`pre-compact.js:105-140` / `session-start.js` / 其他 plan 同步分支）
- **证据**：`tests/test-pace-utils.js` 测试 `getNativePlanPath: 过滤不属于当前项目的 current-native-plan` 实测 FAIL：
  - 期望 `C:/Users/Xiao/AppData/Local/Temp/pace-test-1778742877348-native-plan-filter-project-50i4/docs/plan.md`
  - 实际 `c:/users/xiao/appdata/local/temp/pace-test-1778742877348-native-plan-filter-project-50i4/docs/plan.md`
- **影响**：
  - PreCompact 写入的 `nativePlans` 路径大小写丢失，与其他位置存储的原始大小写路径字符串比较失败
  - bridge hint / Skill 中显示给主 session 的路径与用户实际 cwd 大小写不一致
  - 多 worktree / 多 session 持久化时同一文件可能以不同大小写写入快照
- **建议修复**：
  ```js
  // plans.js:129
  return path.resolve(cwd || process.cwd(), planPath).replace(/\\/g, '/');
  ```
  或新增专用 `displayAbsolutePath()` 工具（只做反斜杠归一不做大小写），与 `normalizePath`（内部比较用）分开
- **与上次审计关联**：上次 [H-4] 指出"Windows 反斜杠泄漏"。本次修复方向正确但**采用了过头的 normalizer**，把大小写也吞了。属于"修对一半"。

### W（Warning） — 8 项

#### W-1：`stop.js` 混合 user-action+execution warning 类型时前缀仍偏 execution

- **位置**：`plugin/hooks/stop.js:298-308`
- **场景**：单次 BLOCK 中 `warningTypes` 同时含 `'user-action'` 与 `'execution'`（如 `unapproved running CHG` + `pending tasks`）
- **后果**：`hasExecutionWarning` 一旦为真即走"请继续执行任务"前缀，无视 `user-action` 信号
- **建议**：前缀使用并集，或将 user-action 拼接到 prefix 第二行
- **与上次审计关联**：[W-8] 的延续——substring 判定已修复，但分类合并策略未覆盖混合场景

#### W-2：`set-artifact-root` 自定义路径冲突检测不做大小写归一

- **位置**：`plugin/hooks/set-artifact-root.js:91-103`
- **场景**：用户传 `--choice /custom/Path` 同时环境变量 `PACEFLOW_ARTIFACT_ROOT=/custom/path`
- **后果**：Windows 下相同路径不同大小写被判为冲突 DENY
- **建议**：自定义路径按 `path.resolve` 后比较；Windows 平台做 case-insensitive 比较；keyword 分支不受影响

#### W-3：`archive-chg` gate 不做 `walkthrough-summary` 必填检查

- **位置**：`plugin/hooks/pre-tool-use/agent-lifecycle-guard.js:384`（close-chg 分支之后缺 `mentionsArchiveChg` 分支）
- **不对称**：close-chg 与 archive-chg 都依赖 `walkthrough-summary` 写 `walkthrough.md` 行；但只有 close-chg 有 hook gate
- **后果**：agent 仍按 instruction 报告 `missing-fields`，不产出错误产物；但与 close-chg 不对称、主 session 反馈延迟一轮
- **建议**：在 `agentLifecyclePromptDenyReason` 增加 `mentionsArchiveChg` 分支，要求 `target` + `walkthrough-summary`

#### W-4：`update-chg action=verify` 文档与模板自相矛盾

- **位置**：
  - `plugin/agents/artifact-writer.md:285-293`（"verify only" 模板只列 `verify-summary` 一个填充字段，暗示必填）
  - `plugin/agent-references/instructions/update-chg.md:57`（字段表"verify-summary（action=verify 时可选...）"）
  - `agent-lifecycle-guard.js:95-103`（无 hook gate）
- **建议**：把 verify only 模板里的 `verify-summary` 注明可选，并与字段表口径一致；或把字段表升为必填+补 hook gate

#### W-5：`promptDeclaredAction` 散文 fallback 仍可能误识 verify

- **位置**：`agent-lifecycle-guard.js:279`
- **正则**：`/(?:^|[\n,，;；])\s*(approve-and-start|update-status|approve|verify)(?=$|[\s,，;；:：])/i`
- **场景**：prompt 同时含 `action: update-status` 与 prose 里 `, verify ...`，prompt 一般走 `promptFieldValue` 优先返回，影响小；但 `## verify only` 模板鼓励主 session 在多段 prompt 出现 "verify" 字样，回归风险增加
- **建议**：进一步收紧（行首/字段化）或与 `promptMentionsVerifyAction` 合并复用判定

#### W-6（新引入）：3 个 skill 文件混用 "skill 根目录" / "skill base directory"
