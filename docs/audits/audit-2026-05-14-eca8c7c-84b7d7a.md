## 8. 审查纪律自省

### 8.1 关键经验

1. **测试失败本身就是高价值发现**：Agent 5 跑测试给出 [H-5.2]，Phase 2 验证后这是 artifact 写保护的关键防御在 Windows 上长期失效的 [C] 级 bug。这种「测试覆盖了但断言落空」的 bug 比「未测试」的 bug 更危险——CI 给人虚假的安全感。**审计纪律应明确：至少一个 agent 必须跑生产测试套件并对比 master HEAD 实际结果。**

2. **路径规范化模式**：本次两个独立 C/H bug 都是同一根因 — Windows 上「输入数据」与「比较基线」斜杠形式不统一。已规范化的位置：roots、displayDir()；遗漏的位置：getNativePlanPath 返回值、bashCommandReferencesArtifact 的 input command 文本。**建议给 pace-utils 加 lint 规则或运行时 invariant：所有路径出口都必须 normalizePath。**

3. **agent stall 防御模式**：今天 4 个 agent stall，都是同一根因——`git diff <range> -- <file>` 在变更超过 ~1500 行时 stdout 流被 watchdog 视为"无进展"。后续审计应预先用 `git log --stat <range> -- <file>` 找 commit 落点，再 `git show <sha>` 逐 commit 看；切忌一次性区间 diff 大文件。

4. **Phase 1 不加约束 + Phase 2 验证 的价值**：Agent 4 报 8 项，Phase 2 路径追踪验证 3 个 [H] 全部成立、5 个 [W]/[I] 也都核实——证明 memory 中「独立审查派单只放通用纪律」纪律保留了发现广度。如果 Phase 1 prompt 中预先排除某些方向，今天 H-1 表格断裂这种「肉眼看 Markdown 源码很容易忽略」的 bug 就会被埋掉。

### 8.2 流程改进建议

- audit skill 应明确 stall 防御："切忌对单文件做 commit 区间 diff，先 `git log --stat` 再 `git show <sha>`"
- audit skill 应增加「跑测试套件并对比 master HEAD」的强制步骤
