## 系统性风险（本轮核心，按优先级）

1. **守卫识别层"单点污染多分支 + bash↔powershell 手工移植不对称"（最高频根因，连续多轮）**: `commandTextLooksMutating` 一个段首锚点缺陷，同时击穿 runtime-control（hardDeny，teammate 也拦）与 artifact-write（denyOrHint）。bash/powershell 两 guard 靠手工移植，每次只补特定 case，反复在前缀/分组/wrapper/别名/反引号/in-place 各方向漏。**根治唯一办法：抽取 bash/powershell 共享的"命令识别层"——mutating 动词表 + wrapper 剥离 + 段首锚点（含 `\n(){}` 与 `&&`/`for…do`）+ 别名集 + 路径 token 规整 + script-engine 识别——为单一模块。** 14 条守卫 P0/P1 塌缩成此 3 个病根。

2. **TEAMMATE 全局 fail-soft 姿态未被文档化（最大系统盲区）**: denyOrHint 在 teammate 下全降级为建议，hardDeny 不降级。**所有 artifact 写保护（含批准/验证 marker 之外的内容写）在 teammate 模式下本就是软的**——这决定了大量 P0/P1 在 teammate 下是"双重 moot"。应：①在 README/REFERENCE 画出"deny vs denyOrHint vs hardDeny 三档 × teammate 是否降级"全表；②评估是否把批准/完整性关键守卫提升为 hardDeny（不随 teammate 降级）。

3. **路径/批准信任链：识别层不规整向上传导成批准门伪造（PU-001 P0）**: `resolveToolFilePath` 不 resolve 绝对路径 → `artifactRelativePathForFile` 漏判 → marker-guard 跳过 → `isChangeApproved` 读 marker 被伪造。**建议把"路径规整"作为所有 artifact 守卫的统一前置**，且批准真相源考虑从 content marker 迁移到受 hook 保护的 frontmatter/runtime state。

4. **fail-open 单点全景 + 最小禁用链（critic）**: `require('./pace-utils')` 失败 15 处 exit 0 + 子守卫 require 在 try 外（ROB-03）+ stdin `null` 崩溃（ROB-01）+ VAULT_PATH 静默降级（PUC-01）+ `.pace/disabled` kill-switch + 绝对路径不 resolve（PU-001），叠加成"一条命令/一个文件即全局放行"链。无单一维度串联，建议专项评估 `.pace/disabled` 创建权限与 require 失败处置统一。

5. **迁移路径数据完整性（MIGV5-01/02）**: `--force` 销毁备份 + 非原子无回滚，在 OneDrive vault 上锁失败概率非低。建议两阶段迁移 + 整循环 try/catch + 已迁移检测。

6. **测试基础设施"修了一半"（TS-01 = W2）**: 共享日志 1MB 截断 × 裸 byte-delta 断言，`:1225` 已加守卫却未推广到 `:5604`/`:5724`。本轮实机触发截断（888→867KB）证明真实。建议 per-test 隔离日志或统一带守卫 delta。
