## 起点：两个问题

1. **bug 是模型能力问题，还是流程问题？**
2. **我们期待 PACEflow + Claude Code 是 100% 不犯错，还是什么？**

以及由此引出的具体设计探问：**除了强制流程，还能用什么办法约束 AI？要不要引入 AI 做质量检验 / 判断主 session 的 AI 是否真的 verified / reviewed？**

---

## 结论 1：bug 归因——是「产生 vs 逃逸」，不是「能力 vs 流程」

「能力 vs 流程」是假二分。正确的拆分是 bug 的**产生**与 bug 的**逃逸**，两者归因不同：

- **bug 的产生 ≈ 内禀**：任何在不确定性下构建复杂系统的主体（人或模型）都会产 bug——需求欠定、后果动手前不可观测、世界在变。不是"不够聪明"，是构建复杂系统的固有属性。
- **模型能力影响某些 bug 的产生率，但解决不了主导失败类**：证据明确——MSR 2026（33,580 PR）显示主导 revert 因是"范围管理与上下文理解"（过度工程、非预期副作用），不是纯功能缺陷；HORIZON 直接下判词"光提升基座模型能力不太可能解决这些失败"。**你最想消灭的那类 bug，恰恰是模型变强也修不掉的那类。**
- **bug 的逃逸 ≈ 流程的事**：流程不阻止产生，它改变"一个缺陷活到 commit/发版的概率"。验证/审计是**捕获**机制，不是**预防**机制。

**归因结论**：bug 产生主要内禀（不确定性 + 能力），bug 逃逸主要靠流程。PACEflow 几乎只作用在第二个——它不让 Claude 更聪明，它让未验证/未批准的改动更难悄悄溜到已提交状态。

---

## 结论 2：我们期待的不是「零缺陷」，是「零沉默失败」+ 按 blast radius 校准

追 100% 不犯错，**既不可能又有害**：

- **不可能**：前沿模型指令遵守 < 100%（IFEval 76.89%），主导失败类不可被能力修复；连确定性 hook 也只 100% 确定"检查发生了"，不保证"检查是对的"，且门本身是代码也有 bug（REVIEWED 空门、`migrate --dryrun` footgun 都是门自己的 bug）。
- **有害**：追零缺陷直接长出自评里的病——19 份审计 / 5014 行却产出 0 P0 / ~60 cosmetic（边际审计抓不到真东西的拐点症状）；还制造虚假安全感（"门多∴安全"），而 METR 证明开发者无法自评流程是否有用。

**正确目标**：让错误的**代价小、寿命短**——
1. 高 blast-radius 的 bug（数据损坏、安全、breaking）→ 重门值得（migrate P0 被发版前抓下是正例）；
2. 低 blast-radius 的 bug（cosmetic、易回滚、内部）→ 接受 + 快速发现即可，上全套 ceremony 是负 ROI；
3. **不重犯**——抓到一次变成测试/correction；
4. **快恢复**——出事能知道改了什么、能回滚（记录/恢复层）。

**一句话**：敌人不是 "bug"，敌人是 **"bug + 健忘 + 无追溯 + 静默漂移"**。PACEflow 的诚实价值主张不该是"更少的 bug"，而是 **"没有沉默的失败"**。

---

## 结论 3：约束 AI 的办法——不是引入 AI judge，~~而是把确定性门的判据从「声明」抬到「证据」~~

> **🔴 撤销说明（2026-06-15 brainstorm 复审）**：本结论后半「把判据从声明抬到证据（evidence gate）」**已撤销**。
> 理由：① 它建立在"让伪造验证变贵 / 不可伪造"的**对抗前提**上，是**范畴错误**——PACEflow 是执行纪律工具，不是质量控制器、不是攻击对抗器（记忆 `paceflow-not-quality-control`）；② "拦截 AI 说谎/伪造证据"是**打不赢的军备竞赛**（防 `echo ok` → AI 可 `&& true` → 无限对抗），且通用项目里 hook **无法确定性区分真测试与假命令**（trilemma：通用 + 确定性 + 零配置 + 不可伪造，四者对"门"无解）；③ 大量验证（UI/手动/读 diff）根本没有命令可捕获，任何"必须有通过命令"的硬门会误杀合法验证。
> **前半「不引入 AI judge」仍成立**（确定性 > LLM-soft，理由见下文）。**正确的纪律边界见「一句话哲学」（已勘误为"结构 vs 内容"）。**

**用 AI 去判断流程合规，无益。** 这是 PACEflow 的founding thesis（项目记忆 `确定性网关 > LLM 软指令`），讨论再次确认：

- AI judge 是概率性的，**当不了确定性门，只能当信号**。把会犯错的 AI 当 gate，等于把当初要逃离的 "statistical hope"（调研 Finding 4）请回来——假放行（缝没补，只糊了层会漏的纸）或假拦截（合法工作被错挡，于是人/AI 开始应付 judge）。
- **递归回归**：AI₂ 判 AI₁，谁判 AI₂？投票降错误率但消不掉；链条最终必须落在确定性检查或人上才能停。

**但当前的 hook 拦的是「声明」，不是「证据」**——这是自评里的 V/R 裂缝：门确定性拦住了"没写 marker"，却拦不住"写了 marker 却没真验证"（`verified-date` / `REVIEWED` 都是 AI 自己写的一句话）。

**补缝的正确方向（零 AI、纯确定性）：把判据从「声明」收紧到「证据」。**

| 门 | 现状（拦声明） | 方向（拦证据，仍纯 hook） |
| --- | --- | --- |
| V（验证） | 接受一句 `verified-date` marker | 要求真实测试产物：命令跑过、退出码 0、输出引用本次改动文件，hook 确定性校验产物存在且匹配 |
| R（审计） | 接受一句 `REVIEWED` marker | 要求真实审计记录 artifact 存在并被链接（如外部 codex/CI review 输出） |

AI 想偷懒跳步，就得伪造一份带退出码的测试产物——**伪造成本远高于写一句 marker**。这仍是 100% "hook 阻拦不合规内容"，只是把"不合规"从**"没声称做"**收紧到**"没有做过的证据"**。

## 天花板：诚实的能力边界

必须承认一个天花板，否则又滑向追零缺陷：

- hook 能确定性校验"一个测试**跑过且通过**"，**校验不了"这个测试有意义（真覆盖到那个 bug）"**。
- 这最后一公里——**测试是否充分、review 是否深入**——**既不能被 hook 确定性保证，也不能被 AI 可靠保证**（它是结论 1 里那层"是否正确"的概率问题）。

**PACEflow 的诚实边界**：确定性保证"真实验证证据存在"，到此为止——**不假装判断验证是否充分**（hook 和 AI 都做不到这件事）。这是对项目记忆 `PACEflow 不做质量控制` 的精化：能确定性强制"验证发生且有证据"，但不强制"验证是好的"。

---

## 一句话哲学（收口）

> **PACEflow 约束的是"该走的步骤被走到 + 顺序正确 + 过程留痕"（结构/编排，可确定性强制），不是"每一步做得真不真、对不对"（内容/真伪——不该做、也做不到）。把力气全押在第一类、对第二类诚实地放手——这才是站得住的护城河。**
>
> _（2026-06-15 勘误：原句作"AI 能不能跳过 / **谎报** / 静默漂移"。"防谎报"属真伪审查，与被撤销的 evidence gate 同属对抗器范畴，已删。纪律工具只管"步骤/顺序/留痕"这层结构，不碰"真伪/对错"这层内容——后者交给人。前者中"偏序门"已在 v7.2.2 落地：verify 需 APPROVED、review 需 VERIFIED。）_

## 候选机制（方向清单，均未进入设计）

按"零 AI、纯确定性、忠于护城河"筛选，未来若要"加强约束"，候选方向是：

1. ~~**V/R 门：声明 → 证据**（本文结论 3）~~ **【2026-06-15 撤销】**——对抗器范畴错误 + 打不赢的军备竞赛 + 通用项目 hook 判不了真伪，与纪律工具身份冲突。详见结论 3 撤销说明。**剩余可落地的纪律强化集中在机制 #2。**
2. **blast-radius 确定性分级**——hook 按改动触及的路径（是否碰 migrate / 门逻辑 / 用户数据路径）确定性判风险等级，据此缩放所需严格度（操作化"按 blast radius 校准"，也对接 lite/full 分档）。
3. **记录 / 恢复层是真护城河**（见自评）——跨 session 上下文恢复、finding/correction 防重犯、可追溯回滚；约束 AI"丢上下文 / 偷懒跳步"靠这一层而非 AI judge。
4. **持续外部审计 > 攒一波集中自审**（见自评 goingForward）——把 codex/另一模型接成定期 gate 产生**信号**（不是 gate 裁决），因为唯一动机无关的发现来自外部模型 + 确定性测试红绿。注意：外部 AI 在此是**信号源**，不是确定性门——与结论 3 一致。
